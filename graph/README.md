# GRAPH

## Алгоритм построения графа

Построение графа основывается на 2 переменных. 

`Edge Keys`  и `node keys` - наборы параметров транзакций, на которых будет основываться группировка.

1. Группируем значения по `edge keys`, каждый набор будет представлять собой некий набор элементов "спуска" по дереву.
2. Каждую группу сортируем по времени транзакции.
3. Внутри каждой группы группируем элементы по `node keys`. Каждая подгруппа будет представлять из себя вершину нашего графа.

### Свойства

1. Такой граф всегда будет являться деревом, что позволяет применять большее количество подходов в анализе данных и обучении моделей.
2. Гибкая структура графа позволяет менять акцент в обучении модели без изменения общего пайплайна.
3. Граф отражает временную зависимость транзакций и сразу несет в себе элемент обучения на последовательностях, что необходимо в контексте анализа транзакций.

## GNN

### Структура модели

В эксперименте используется гибридная архитектура, которая объединяет:
1. **табличное представление транзакции**;
2. **графовое представление локального контекста**;
3. **агрегацию по дереву транзакций**, построенному на основе `CustomerId` и `ProductCategory`.

#### Входные признаки транзакции

Для каждой транзакции формируется три группы признаков:

- **Категориальные признаки**:
  - `BatchId`
  - `AccountId`
  - `SubscriptionId`
  - `CustomerId`
  - `CurrencyCode`
  - `ProviderId`
  - `ProductId`
  - `ProductCategory`
  - `ChannelId`

  Каждый категориальный признак кодируется через отдельный `Embedding`.

- **Числовые признаки**:
  - `Amount`
  - `Value`
  - `CountryCode`
  - `PricingStrategy`

  Перед обучением числовые признаки стандартизуются по статистикам train-выборки.

- **Временные признаки**:
  - час транзакции;
  - день недели.

  Они извлекаются из `TransactionStartTime` и нормализуются в диапазон `[0, 1]`.

#### Encoder транзакции

Блок `TxEncoder`:
- получает embeddings категориальных признаков;
- конкатенирует их с числовыми и временными признаками;
- пропускает через двухслойный MLP;
- формирует скрытое представление транзакции размерности `hidden=64`.

Таким образом, каждая транзакция получает векторное представление `e_tx`.

#### Построение признаков вершин графа

После построения графа каждая транзакция сопоставляется вершине дерева через `tx_node`.

Для каждой вершины используются два источника информации:
1. **агрегированное представление транзакций**, попавших в эту вершину:
   - берется среднее по эмбеддингам транзакций (`scatter_mean`);
2. **сырой признак вершины**:
   - размер вершины (`node.size`), то есть число транзакций в ней.

Далее:
- сырой признак вершины кодируется через `node_enc`;
- объединяется с агрегированным представлением транзакций;
- пропускается через `node_fuse`.

В результате получается начальное скрытое состояние вершины графа.

#### Графовая часть

Для распространения информации по дереву используются `gnn_layers=2` слоя `GraphSAGE`.

Каждый слой:
- сохраняет информацию о текущей вершине (`self`);
- агрегирует среднее по соседям (`neighbor mean`);
- объединяет оба источника через линейные преобразования и `ReLU`.

Это позволяет вершине учитывать не только собственные транзакции, но и ближайший структурный контекст внутри дерева.

#### Голова классификации

Финальное решение по транзакции принимается на основе:
- локального представления самой транзакции `e_tx`;
- графового контекста вершины `h[tx_node]`.

Эти два вектора конкатенируются и подаются в `tx_head`, который выдает логит мошенничества для каждой транзакции.

#### Обучение

Модель обучается как бинарный классификатор:
- функция потерь: `BCEWithLogitsLoss`;
- для компенсации дисбаланса классов используется `pos_weight`;
- оптимизатор: `Adam`.

Во время обучения после каждой эпохи считается качество на валидации, а лучший checkpoint выбирается по **`val PR-AUC`**.
После этого отдельно подбирается оптимальный `threshold` по валидационной выборке с целевой метрикой **`F0.5`**.

### Лучшие метрики

Подбор гиперпараметров выполнялся через явный `grid search` по:
- `learning rate`;
- `threshold`.

Фиксированные параметры модели:
- `epochs = 125`
- `hidden = 64`
- `gnn_layers = 2`

#### Лучшая конфигурация на валидации

По результатам перебора лучшей оказалась следующая комбинация:

- **learning rate**: `1.2e-02`
- **threshold**: `0.41`
- **best epoch (по val PR-AUC)**: `8`

Метрики на валидационной выборке для лучшей конфигурации:

- **F0.5**: `0.696203`
- **F2**: `0.723684`
- **Precision**: `0.687500`
- **Recall**: `0.733333`
- **Accuracy**: `0.998746`
- **ROC-AUC**: `0.921412`
- **PR-AUC**: `0.534844`

#### Финальные метрики на тестовой выборке

После фиксации лучшей пары `(lr, threshold)` модель была оценена на test.

#### Интерпретация результата

Итоговая модель показывает:
- очень высокое качество ранжирования (`ROC-AUC`, `PR-AUC`);
- высокий recall при хорошем precision;
- устойчивый результат как на уровне отдельных транзакций, так и на уровне клиентов.

Это означает, что графовая структура действительно добавляет полезный контекст для выявления мошеннических операций.

### Выводы

Графовый подход оказался удачным для задачи детекции мошенничества.

#### Что сработало хорошо

1. **Структурирование данных в виде дерева**
   
   Построение графа по `CustomerId` с разбиением по `ProductCategory` позволяет явно учитывать:
   - поведение клиента;
   - последовательность его действий во времени;
   - локальный контекст внутри группы транзакций.

2. **Комбинация табличных и графовых признаков**
   
   Модель использует одновременно:
   - сырые признаки транзакции;
   - временные признаки;
   - категориальные embeddings;
   - графовый контекст вершины.

   За счет этого классификатор принимает решение не только по самой транзакции, но и по ее окружению.

3. **GraphSAGE хорошо подходит для данной постановки**
   
   Графовые слои позволяют распространять информацию между связанными вершинами дерева и усиливать локальные паттерны мошеннического поведения.

4. **Подбор threshold оказался критически важен**
   
   Качество модели заметно зависит от порога бинаризации.  
   Простое значение `0.5` не является универсально лучшим, поэтому отдельный подбор `threshold` по валидации дает ощутимый прирост в целевой метрике `F0.5`.

#### Практические наблюдения

- Лучшая эпоха для выбранного `learning rate` нашлась уже на **8-й эпохе**.
- Это говорит о том, что:
  - модель довольно быстро выходит на хороший режим;
  - дальнейшее обучение не обязательно улучшает валидационное качество;
  - для такого пайплайна имеет смысл рассмотреть **early stopping**.

#### Итог

Разработанный GNN-пайплайн показал, что граф, построенный по бизнес-логике транзакций, действительно помогает улучшить качество fraud detection.

Основной вывод:
- **графовая структура + эмбеддинги транзакций + подбор порога классификации** дают сильное и интерпретируемое решение для задачи обнаружения мошенничества.

В дальнейшем качество можно улучшать за счет:
- расширения набора `edge_keys` и `node_keys`;
- более богатых признаков вершины;
- сравнения с альтернативными GNN-архитектурами.
