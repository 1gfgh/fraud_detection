# Transformers

В данной папке находятся эксперименты с трансформерными моделями для задачи обнаружения мошеннических транзакций. Основная идея блока — проверить, насколько архитектуры на основе self-attention подходят для анализа последовательностей транзакций клиентов и могут ли они конкурировать с рекуррентными сетями.

## Содержимое папки

- `transformer_sequence_comparison.ipynb` — первичное сравнение нескольких трансформерных архитектур;
- `mean_tuning.ipynb` — расширенный подбор гиперпараметров для модели `Transformer Mean`;
- `meanmax_tuning.ipynb` — расширенный подбор гиперпараметров для модели `Transformer MeanMax`.

## Зачем в проекте нужны трансформеры

Трансформеры полезны для работы с последовательностями, когда важно учитывать взаимосвязи между разными элементами не только по соседству, но и на дальних расстояниях.

В контексте fraud detection это означает, что модель может анализировать не просто текущую транзакцию, а всю историю операций клиента как единую последовательность и искать:

- аномальные сочетания транзакций;
- резкие смены паттернов поведения;
- нетипичные временные и структурные зависимости;
- важные события, удалённые друг от друга по времени.

В отличие от RNN, LSTM и GRU, трансформер использует механизм **self-attention**, который позволяет напрямую сопоставлять все элементы последовательности друг с другом.

## Подготовка данных

Во всех ноутбуках этого блока использовалась единая схема подготовки данных:

- транзакции загружались из `train.csv`, `val.csv`, `test.csv`;
- выполнялась инженерия признаков;
- для каждого клиента формировалась последовательность транзакций;
- максимальная длина последовательности ограничивалась **100 транзакциями**;
- короткие последовательности дополнялись нулями;
- признаки нормализовывались;
- итоговая классификация выполнялась **на уровне клиента**.

Таким образом, каждая модель получала на вход последовательность транзакций одного клиента и должна была предсказать, есть ли у клиента мошенническая активность.

## Краткая теория

### Positional Encoding
Так как трансформер сам по себе не знает порядок элементов, в модель добавлялось **позиционное кодирование**, чтобы сохранить информацию о порядке транзакций в последовательности.

### Transformer Encoder
Основной блок модели — `TransformerEncoder`, который строит контекстные представления шагов последовательности с помощью multi-head self-attention и feed-forward слоёв.

### Способы агрегации последовательности

В проекте были рассмотрены несколько способов получить итоговое представление клиента из последовательности скрытых состояний:

- **Mean pooling** — усреднение всех hidden states по времени;
- **CLS token** — добавление специального классификационного токена, чьё представление используется для финального решения;
- **Attention pooling** — обучаемое взвешивание шагов последовательности;
- **MeanMax pooling** — объединение среднего и максимума, чтобы сохранить и общий паттерн, и наиболее сильные всплески в последовательности.

## Этап 1. Сравнение трансформерных архитектур — `transformer_sequence_comparison.ipynb`

В ноутбуке `transformer_sequence_comparison.ipynb` сравнивались **4 архитектуры**:

- `TransformerMean`
- `TransformerCLS`
- `TransformerAttention`
- `TransformerMeanMax`

### Архитектуры моделей

Все модели имели общую основу:

- входную линейную проекцию признаков;
- positional encoding;
- `TransformerEncoder`;
- dropout;
- классификационную голову.

Главное различие между ними заключалось в способе агрегирования последовательности:

- `TransformerMean` — среднее по всем шагам;
- `TransformerCLS` — использование специального `CLS`-токена;
- `TransformerAttention` — attention pooling;
- `TransformerMeanMax` — конкатенация mean pooling и max pooling.

## Результаты сравнения на уровне клиента

| TransformerCLS | 0.9862 | 0.8063 | 0.7778 | 0.7778 | 0.7778 | 0.7778 | 738 | 2 | 2 | 7 |

## Выводы по сравнению

По итогам первичного сравнения можно выделить несколько наблюдений:

### Лучшая модель по `F0.5`
Лучший результат по основной метрике показала **TransformerMean**:

- **F0.5 = 0.8163**

Так как `F0.5` была ключевой метрикой проекта, эта модель стала очень важным кандидатом для дальнейшего тюнинга.

### Лучшая модель по `PR-AUC`
Лучший результат по `PR-AUC` показала **TransformerMeanMax**:

- **PR-AUC = 0.8677**

То есть MeanMax-вариант оказался сильным с точки зрения качества ранжирования fraud-клиентов.

### Почему для тюнинга были выбраны Mean и MeanMax

- `TransformerMean` был лучшим по основной метрике `F0.5`;
- `TransformerMeanMax` был лучшим по `PR-AUC`.

Именно поэтому эти две архитектуры были выбраны для расширенного подбора гиперпараметров с помощью Optuna.

## Этап 2. Тюнинг Transformer Mean — `mean_tuning.ipynb`

### Архитектура модели

В `mean_tuning.ipynb` использовалась модель `TransformerMeanClassifier`, включающая:

- входную проекцию признаков в пространство `d_model`;
- positional encoding;
- стек encoder-слоёв трансформера;
- `LayerNorm`;
- **mean pooling** по всей последовательности;
- усиленную classifier head.

Такой вариант делает ставку на усреднённое представление истории транзакций клиента.

### Что подбиралось в Optuna

Для модели автоматически подбирались:

- `d_model` — размер внутреннего представления последовательности;
- `nhead` — число attention-heads;
- `num_layers` — число encoder-слоёв;
- `dim_feedforward` — размер feed-forward блока внутри encoder;
- `dropout` — регуляризация;
- `learning_rate` — скорость обучения;
- `weight_decay` — L2-регуляризация;
- `optimizer_type` — выбор оптимизатора (`adam`, `adamw`, `sgd`);
- `criterion_type` — функция потерь (`ce`, `weighted_ce`, `focal`);
- `scheduler_type` — схема изменения learning rate;
- `batch_size` — размер батча;
- `clip_grad` — ограничение нормы градиента;
- при некоторых вариантах также:
  - `momentum` — для SGD;
  - `fraud_weight` — вес fraud-класса;
  - `alpha_fraud`, `gamma` — параметры focal loss.

### Что означают эти параметры

- `d_model` определяет “ширину” представления каждого шага последовательности;
- `nhead` задаёт, на сколько независимых attention-проекций разбивается анализ;
- `num_layers` отвечает за глубину encoder-части;
- `dim_feedforward` задаёт размер внутреннего преобразования внутри слоя трансформера;
- `clip_grad` помогает стабилизировать обучение.

### Лучшие гиперпараметры Transformer Mean

По итогам **50 trials** лучший результат на валидации составил:

- **Best validation F0.5 = 0.8772**

Лучшая конфигурация:

- `d_model = 96`
- `nhead = 4`
- `num_layers = 4`
- `dim_feedforward = 1024`
- `dropout = 0.3`
- `learning_rate = 0.000066`
- `weight_decay = 0.000033`
- `optimizer_type = adamw`
- `criterion_type = ce`
- `scheduler_type = cosine`
- `batch_size = 16`
- `clip_grad = 2.0`

### Итоговые результаты Transformer Mean после тюнинга

На тесте модель показала:

- `TN = 739`
- `FP = 1`
- `FN = 3`
- `TP = 6`

Ключевые метрики на уровне клиента:

- **F0.5 = 0.8108**
- **PR-AUC = 0.8137**
- **ROC-AUC = 0.9658**
- **Precision = 0.8571**
- **Recall = 0.6667**
- **F1 = 0.7500**

Оптимальный порог, найденный по validation:

- **threshold = 0.42**

### Вывод по Transformer Mean

После тюнинга модель сохранила очень сильный результат по основной метрике `F0.5` и показала особенно высокую precision при минимальном числе ложноположительных срабатываний (`FP = 1`).

## Этап 3. Тюнинг Transformer MeanMax — `meanmax_tuning.ipynb`

### Архитектура модели

В `meanmax_tuning.ipynb` использовалась модель `TransformerMeanMaxClassifier`, включающая:

- входную проекцию признаков;
- positional encoding;
- `TransformerEncoder`;
- `LayerNorm`;
- **mean pooling**;
- **max pooling**;
- объединение этих двух представлений;
- усиленную classifier head.

Идея MeanMax-подхода состоит в том, чтобы одновременно использовать:

- усреднённую информацию по всей последовательности;
- наиболее сильные локальные активации.

### Что подбиралось в Optuna

Для MeanMax-модели подбирались те же основные параметры:

- `d_model`
- `nhead`
- `num_layers`
- `dim_feedforward`
- `dropout`
- `learning_rate`
- `weight_decay`
- `optimizer_type`
- `criterion_type`
- `scheduler_type`
- `batch_size`
- `clip_grad`

Дополнительно, в зависимости от выбранной функции потерь или оптимизатора, подбирались:

- `fraud_weight`
- `alpha_fraud`
- `gamma`
- `momentum`

### Лучшие гиперпараметры Transformer MeanMax

По итогам **50 trials** лучший результат на валидации составил:

- **Best validation F0.5 = 0.9836**

Лучшая конфигурация:

- `d_model = 192`
- `nhead = 8`
- `num_layers = 2`
- `dim_feedforward = 384`
- `dropout = 0.2`
- `learning_rate = 0.000649`
- `weight_decay = 0.000007`
- `optimizer_type = adamw`
- `criterion_type = weighted_ce`
- `scheduler_type = cosine`
- `batch_size = 16`
- `clip_grad = 1.5`
- `fraud_weight = 11.250269`

### Итоговые результаты Transformer MeanMax после тюнинга

На тесте модель показала:

- `TN = 737`
- `FP = 3`
- `FN = 1`
- `TP = 8`

Ключевые метрики на уровне клиента:

На валидации модель выглядела очень сильно, но на тесте итоговый `PR-AUC` оказался заметно ниже ожидаемого. При этом модель сохранила высокий recall и хорошее общее качество по ROC-AUC.

## Финальный вывод

Блок `transformers` показал, что attention-модели действительно применимы к задаче fraud detection на клиентских последовательностях.

Главный результат этого раздела:

- трансформеры способны давать конкурентное качество;
- способ pooling заметно влияет на поведение модели;
- `mean pooling` в данной постановке оказался стабильнее и полезнее, чем `meanmax pooling` после полного тюнинга;
- наиболее сильной моделью блока стала **tuned Transformer Mean**.
