# Baseline

В этой папке находятся эксперименты с бустинговыми моделями для задачи обнаружения мошенничества. Основная модель — `CatBoostClassifier`.

## Содержимое

- `baseline.ipynb` — базовый CatBoost с фиксированными гиперпараметрами;
- `catboost_pro.ipynb` — CatBoost с подбором гиперпараметров через Optuna.

## Данные

Ноутбуки используют уже подготовленные файлы из папки `data`:

- `../data/train_clean.csv`
- `../data/val_clean.csv`
- `../data/test_clean.csv`

Предобработка признаков теперь выполняется централизованно в `data/cleanup.ipynb`. В baseline-ноутбуках данные только считываются, после чего из них формируются матрицы для обучения.

Размеры выборок:

- `train_clean.csv`: **57 364** строки, **24** колонки;
- `val_clean.csv`: **19 162** строки, **24** колонки;
- `test_clean.csv`: **19 136** строк, **24** колонки.

Доля мошеннических транзакций:

- `train`: **0.20%**
- `val`: **0.21%**
- `test`: **0.20%**

Задача остается сильно несбалансированной.

## Признаки

В модель подается **22 признака**. `CustomerId` используется только для агрегации предсказаний на уровень клиента, а `FraudResult` является целевой переменной.

- `tx_dayofweek`

## Оценка

Модели обучаются на транзакционном уровне, но финальная оценка считается на уровне клиента.

Агрегация выполняется так:

- истинная метка клиента — максимум `FraudResult` по его транзакциям;
- score клиента — максимум предсказанного score по его транзакциям.

В текущем запуске `test_clean.csv` после агрегации содержит:

- **749** клиентов;
- **11** мошеннических клиентов;
- **738** честных клиентов.

## Вывод

Оба ноутбука корректно используют новые подготовленные датасеты и строят матрицы признаков одинакового формата: `X_train`, `X_val` и `X_test` содержат по **22** признака.

`catboost_pro.ipynb` улучшил основную метрику `F0.5` и полностью убрал пропущенных мошеннических клиентов на тесте (`FN = 0`), но сделал это за счет низкого подобранного порога **0.04**. Это стоит учитывать при интерпретации: validation очень маленький по числу fraud-клиентов, поэтому идеальный `F0.5 = 1.0` на validation не нужно считать надежной гарантией обобщения.
