# Recurrent Models

В данной папке находятся эксперименты с рекуррентными нейронными сетями для задачи обнаружения мошеннических транзакций. Основная идея этого блока — проверить, дают ли последовательностные модели преимущество по сравнению с обычными табличными подходами за счёт учёта порядка транзакций во времени.

## Содержимое папки

- `compare.ipynb` — первичное сравнение нескольких рекуррентных архитектур;
- `gru_tuning.ipynb` — расширенный подбор гиперпараметров для улучшенной GRU-модели;
- `bilstm_tuning.ipynb` — расширенный подбор гиперпараметров для улучшенной BiLSTM-модели.

## Зачем в проекте нужны рекуррентные сети

В задаче fraud detection важно не только смотреть на отдельную транзакцию, но и учитывать поведение клиента как последовательность операций.  
Например, мошенничество может проявляться не в одном признаке, а в паттерне:

- в необычном порядке действий;
- в резкой смене типа операций;
- в аномальной активности по времени;
- в повторяющихся или резко отличающихся транзакциях.

Рекуррентные сети как раз предназначены для работы с последовательностями.

### Кратко о моделях

- **RNN** — базовая рекуррентная сеть, умеет учитывать порядок элементов, но хуже работает с длинными зависимостями.
- **LSTM** — более сложная архитектура с механизмом памяти, лучше удерживает важную информацию на длинных последовательностях.
- **GRU** — упрощённый вариант LSTM, обычно работает быстрее и проще в обучении.
- **BiLSTM / BiGRU** — двунаправленные версии, которые анализируют последовательность сразу в двух направлениях и могут извлекать более богатое представление.

Для транзакционных данных это полезно, потому что модель получает возможность учитывать контекст всей последовательности клиента, а не только отдельную запись.

## Общая логика экспериментов

Работа с recurrent-моделями была организована в два этапа:

1. сначала сравнивались несколько разных архитектур в базовой постановке;
2. затем две наиболее сильные модели были выбраны для углублённого тюнинга с помощью Optuna.

Такой подход позволил сначала определить перспективные архитектуры, а уже потом тратить вычислительные ресурсы на подбор гиперпараметров только для лучших кандидатов.

## Этап 1. Первичное сравнение моделей — `compare.ipynb`

В ноутбуке `compare.ipynb` сравнивались **6 моделей**:

- `RNN`
- `LSTM`
- `BiLSTM`
- `GRU`
- `ImprovedGRU`
- `ImprovedLSTM`

### Что отличало улучшенные версии

Улучшенные архитектуры (`ImprovedGRU`, `ImprovedLSTM`) были существенно богаче базовых вариантов и включали:

- входную проекцию признаков;
- bidirectional-рекуррентный слой;
- `LayerNorm`;
- `attention pooling`;
- `mean pooling`;
- `max pooling`;
- более сильную классификационную голову.

Идея заключалась в том, чтобы не ограничиваться только последним hidden state, а собрать более полное представление о последовательности клиента.

## Выводы по сравнению

По итогам первичного этапа можно выделить два главных наблюдения:

### 1. Лучшая модель по `F0.5`
Лучший результат по основной метрике показала **BiLSTM**:

- **F0.5 = 0.7895**

Это важно, потому что именно `F0.5` являлась ключевой метрикой проекта и сильнее акцентировала внимание на precision.

### 2. Лучшая модель по `PR-AUC`
Лучший результат по `PR-AUC` показала **GRU**:

- **PR-AUC = 0.8678**

Также она продемонстрировала высокий recall и сильное общее качество.

### Почему для тюнинга были выбраны именно GRU и BiLSTM

- **BiLSTM** оказалась лучшей по `F0.5` и достигла идеального recall (`1.0`) в базовом сравнении;
- **GRU** показала лучший `PR-AUC` и сильный баланс по ключевым метрикам.

Поэтому именно эти две архитектуры были выбраны для дальнейшего углублённого тюнинга.

## Этап 2. Тюнинг BiLSTM — `bilstm_tuning.ipynb`

В ноутбуке `bilstm_tuning.ipynb` использовалась улучшенная bidirectional LSTM-архитектура.

### Архитектура модели

Модель включала:

- входную линейную проекцию признаков;
- bidirectional LSTM;
- `LayerNorm` на выходах LSTM;
- `attention pooling`;
- усреднение по времени (`mean pooling`);
- максимум по времени (`max pooling`);
- объединение нескольких представлений последовательности;
- многослойную classifier head.

То есть модель использовала не только последний hidden state, а сразу несколько способов агрегировать информацию о последовательности клиента.

### Что подбиралось в Optuna

Для BiLSTM автоматически подбирались:

- `hidden_dim` — размер скрытого состояния, то есть объём внутренней памяти сети;
- `num_layers` — число LSTM-слоёв;
- `dropout` — уровень регуляризации;
- `learning_rate` — скорость обучения;
- `weight_decay` — L2-регуляризация;
- `optimizer_type` — тип оптимизатора (`adam`, `adamw`, `sgd`);
- `criterion_type` — функция потерь (`ce`, `weighted_ce`, `focal`);
- `scheduler_type` — стратегия изменения learning rate;
- `batch_size` — размер мини-батча;
- `fraud_weight` — дополнительный вес fraud-класса для weighted cross-entropy;
- `alpha_fraud`, `gamma` — параметры focal loss;
- при использовании SGD также мог подбираться `momentum`.

### Что это даёт

Такой подбор позволял найти не просто лучшую архитектуру, а наиболее удачное сочетание:

- ёмкости модели;
- регуляризации;
- режима оптимизации;
- способа борьбы с дисбалансом классов.

### Лучшие гиперпараметры BiLSTM

По итогам **50 trials** лучший результат на валидации составил:

- **Best validation F0.5 = 0.9434**

Лучшая конфигурация:

- `hidden_dim = 96`
- `num_layers = 2`
- `dropout = 0.2`
- `learning_rate = 0.000295`
- `weight_decay = 0.000540`
- `optimizer_type = adamw`
- `criterion_type = weighted_ce`
- `scheduler_type = none`
- `batch_size = 32`
- `fraud_weight = 2.221601`

### Итоговые результаты BiLSTM после тюнинга

На тесте модель показала:

- `TN = 737`
- `FP = 3`
- `FN = 1`
- `TP = 8`

Ключевые метрики:

- **F0.5 = 0.7547**
- **PR-AUC = 0.8656**
- **ROC-AUC = 0.9973**
- **Precision = 0.7273**
- **Recall = 0.8889**
- **F1 = 0.8000**

### Вывод по BiLSTM

Тюнинг сохранил сильное качество модели, особенно по:

- `PR-AUC`
- `Recall`
- `ROC-AUC`

BiLSTM осталась очень сильной моделью для сценариев, где важно находить как можно больше мошенников и при этом сохранять достойную точность.

## Этап 3. Тюнинг GRU — `gru_tuning.ipynb`

В ноутбуке `gru_tuning.ipynb` использовалась улучшенная bidirectional GRU-архитектура.

### Архитектура модели

Структура модели была аналогична улучшенной LSTM-версии и включала:

- входную проекцию;
- bidirectional GRU;
- `LayerNorm`;
- `attention pooling`;
- `mean pooling`;
- `max pooling`;
- объединение нескольких представлений последовательности;
- усиленную классификационную голову.

Такой подход позволял GRU извлекать более информативное представление последовательности транзакций клиента.

### Что подбиралось в Optuna

Для GRU подбирались:

- `hidden_dim` — размер скрытого состояния;
- `num_layers` — число GRU-слоёв;
- `dropout` — регуляризация;
- `learning_rate` — шаг обучения;
- `weight_decay` — L2-регуляризация;
- `optimizer_type` — выбор оптимизатора;
- `criterion_type` — функция потерь;
- `scheduler_type` — схема изменения learning rate;
- `batch_size` — размер батча;
- `clip_grad` — ограничение нормы градиента для устойчивого обучения;
- `fraud_weight` — вес fraud-класса при weighted cross-entropy;
- `alpha_fraud`, `gamma` — параметры focal loss;
- `momentum` — параметр SGD.

### Роль гиперпараметров

Здесь особенно важно было подобрать не только архитектуру, но и устойчивый режим обучения, потому что sequence-модели на несбалансированных данных чувствительны к:

- размеру шага обучения;
- глубине сети;
- батчу;
- выбору функции потерь;
- стабилизации градиентов.

### Лучшие гиперпараметры GRU

По итогам **50 trials** лучший результат на валидации составил:

- **Best validation F0.5 = 0.9184**

Лучшая конфигурация:

- `hidden_dim = 32`
- `num_layers = 4`
- `dropout = 0.2`
- `learning_rate = 0.000614`
- `weight_decay = 0.000004`
- `optimizer_type = adamw`
- `criterion_type = ce`
- `scheduler_type = cosine`
- `batch_size = 128`
- `clip_grad = 1.0`

### Итоговые результаты GRU после тюнинга

На тесте модель показала:

- `TN = 738`
- `FP = 2`
- `FN = 3`
- `TP = 6`

Ключевые метрики:

- **F0.5 = 0.7317**
- **PR-AUC = 0.7483**
- **ROC-AUC = 0.9715**
- **Precision = 0.7500**
- **Recall = 0.6667**
- **F1 = 0.7059**

### Вывод по GRU

После тюнинга GRU получила:

- высокую precision;
- низкое число ложноположительных срабатываний (`FP = 2`);
- сильный контроль качества по основной метрике.

Однако по итоговым значениям `F0.5` и `PR-AUC` tuned GRU уступила tuned BiLSTM.

## Основной итог по recurrent-моделям

Рекуррентные сети показали, что учёт последовательности транзакций действительно полезен для задачи fraud detection.

По итогам всех экспериментов:

- **лучшей recurrent-моделью по основной метрике `F0.5` стала BiLSTM**;
- **GRU также показала сильные результаты**, но в финальной версии уступила BiLSTM по `F0.5` и `PR-AUC`;
- использование bidirectional-архитектуры, pooling-механизмов и attention действительно помогло улучшить качество по сравнению с более простыми моделями.

## Финальный вывод

Блок `recurrent` показал, что последовательностные модели являются сильным направлением для обнаружения мошенничества в банковских транзакциях.

Наиболее удачной моделью в этом разделе можно считать **tuned BiLSTM**, поскольку она обеспечила:

- высокий `F0.5`;
- высокий `PR-AUC`;
- сильный recall;
- небольшое число ошибок на редком fraud-классе.

Таким образом, если в рамках проекта требуется именно сильная sequence-модель для анализа транзакционной истории клиента, то основным результатом этого блока является **BiLSTM после подбора гиперпараметров**.
