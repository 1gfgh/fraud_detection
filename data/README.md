# Data

В данной папке находятся исходные и подготовленные данные для задачи обнаружения мошеннических транзакций, а также ноутбук с логикой очистки и разбиения выборки.

## Содержимое папки

- `cleanup.ipynb` — подготовка данных, анализ структуры датасета и разбиение на `train / val / test`;
- `full.csv` — полный датасет, используемый как исходный источник данных;
- `train_raw.csv`, `val_raw.csv`, `test_raw.csv` — выборки после разбиения по клиентам, сохранённые со всеми исходными признаками;
- `train.csv`, `val.csv`, `test.csv` — очищенные выборки после удаления части служебных и дублирующих признаков.

## Исходные данные

Исходный датасет содержит:

- **95 662 транзакции**;
- **3 742 уникальных клиента**;
- **16 исходных признаков**.

Целевая переменная — `FraudResult`, где отмечаются мошеннические транзакции.

Общий уровень мошенничества в данных невысокий:
- fraud rate по транзакциям: **0.0020**;
- число мошеннических транзакций: **193**;
- число клиентов, у которых встречалось мошенничество: **54**.

Таким образом, задача характеризуется выраженным дисбалансом классов, что необходимо учитывать при дальнейшем обучении и оценке моделей.

## Что было сделано в `cleanup.ipynb`

В ноутбуке `cleanup.ipynb` были выполнены следующие шаги:

1. загрузка и первичный анализ исходного датасета;
2. проверка типов данных и пропусков;
3. анализ распределения транзакций и мошенничества на уровне клиентов;
4. разбиение данных на обучающую, валидационную и тестовую части;
5. сохранение двух версий выборок:
   - `raw`-версий со всеми признаками;
   - очищенных версий после удаления части столбцов.

Пропусков в исходном датасете обнаружено не было.

## Логика разбиения

Ключевой особенностью подготовки данных стало разбиение **по клиентам (`CustomerId`)**, а не по отдельным транзакциям.

Это было сделано для того, чтобы транзакции одного и того же клиента не попадали одновременно в разные части выборки. В задаче fraud detection это особенно важно, поскольку иначе возникает утечка информации: модель может косвенно запомнить особенности клиента на обучении и показать искусственно завышенное качество на тесте.

Разбиение выполнялось в пропорции:

- **train** — 60% клиентов;
- **val** — 20% клиентов;
- **test** — 20% клиентов.

Итоговое распределение клиентов:

- `train`: **2245** клиентов;
- `val`: **748** клиентов;
- `test`: **749** клиентов.

Пересечений между выборками нет:

- `Train ∩ Val = 0`;
- `Train ∩ Test = 0`;
- `Val ∩ Test = 0`.

Таким образом, каждая группа содержит собственное непересекающееся множество клиентов.

## Особенности баланса

В ноутбуке учитывалось распределение мошенничества на уровне клиента. Для этого рассчитывались клиентские характеристики:

- число мошеннических транзакций клиента;
- fraud rate клиента;
- общее число транзакций клиента.

Данные сильно разрежены по мошенничеству: только **54 клиента** из **3 742** имеют хотя бы одну fraudulent-транзакцию. Поэтому простое квантильное разбиение клиентов по `fraud_count` или `fraud_rate` не даёт устойчивых страт: большинство клиентов имеет нулевые значения.

В итоговой версии разбиение было сделано иначе:

- fraud-клиенты распределялись отдельно, чтобы сбалансировать число fraud-клиентов и fraud-транзакций;
- non-fraud клиенты дополнительно балансировались по числу транзакций;
- каждый клиент всё равно попадал только в один split.

Fraud rate по транзакциям после разбиения составил:

- `train`: **0.0020** (114 fraud);
- `val`: **0.0021** (40 fraud);
- `test`: **0.0020** (39 fraud);
- исходный датасет: **0.0020** (193 fraud).

Fraud rate по клиентам:

- `train`: **0.0053**;
- `val`: **0.0057**;
- `test`: **0.0060**;
- исходный датасет: **0.0056**.

Число клиентов с мошенничеством:

- `train`: **32**;
- `val`: **11**;
- `test`: **11**.

Таким образом, разбиение сохраняет близкий fraud rate по транзакциям и достаточно ровно распределяет редких fraud-клиентов между выборками.

## Размеры выборок

Итоговые размеры по транзакциям:

- `train`: **57 364** строк (**60.0%**);
- `val`: **19 162** строки (**20.0%**);
- `test`: **19 136** строк (**20.0%**).

Разбиение выполнялось по клиентам, но при подборе split дополнительно учитывалось число транзакций на клиента. Поэтому итоговое распределение по строкам получилось близким к целевым **60/20/20**.

По данным анализа:

- среднее число транзакций на клиента: **25.56**;
- медиана: **7**;
- максимум: **4091**.

Из-за сильной неоднородности клиентов по числу транзакций равномерное разбиение только по клиентам могло бы дать заметный перекос по строкам. В текущей версии этот перекос дополнительно контролируется.

## Raw- и cleaned-версии данных

После разбиения были сохранены две версии файлов.

### Raw-файлы

Файлы:

- `train_raw.csv`
- `val_raw.csv`
- `test_raw.csv`

содержат все исходные признаки и используются как промежуточный результат после разбиения.

### Очищенные файлы

На следующем этапе из raw-версий были удалены признаки:

- `BatchId`
- `SubscriptionId`
- `AccountId`
- `CurrencyCode`
- `CountryCode`
- `Value`

После очистки осталось **10 колонок**:

- `TransactionId`
- `CustomerId`
- `ProviderId`
- `ProductId`
- `ProductCategory`
- `ChannelId`
- `Amount`
- `TransactionStartTime`
- `PricingStrategy`
- `FraudResult`

Финальные размеры очищенных файлов:

- `train.csv` — **57 364 строки, 10 колонок**;
- `val.csv` — **19 162 строки, 10 колонок**;
- `test.csv` — **19 136 строк, 10 колонок**.

Эти файлы содержат очищенную транзакционную таблицу и сохраняют исходные категориальные значения, идентификатор клиента, время транзакции и целевую переменную.

## Итог

В результате этапа подготовки были получены три непересекающиеся по клиентам выборки для обучения, валидации и тестирования моделей.

Такой способ подготовки данных позволил:

- исключить утечку информации между выборками;
- сохранить близкое распределение целевой переменной;
- получить корректную основу для дальнейшего сравнения различных подходов к fraud detection.
